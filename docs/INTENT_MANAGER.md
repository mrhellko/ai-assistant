# Intent Manager

`IntentManager` - единственная точка классификации пользовательского текста перед
передачей запроса в доменные модули. Сейчас он живет в
`services/assistant/app/agents/intent_manager.py`.

## Задача

Модуль принимает текст пользователя, текущую дату/время, таймзону и минимальный
контекст, затем возвращает строго структурированный JSON. Backend не пытается
угадывать намерение локальными шаблонами фраз.

## Поддерживаемые intent

Список intent хранится в таблице `intent_definitions` и используется при сборке
system prompt. Пока в проекте нет Alembic, таблица наполняется deterministic
bootstrap-кодом при старте приложения. После ввода Alembic эти записи должны
переехать в миграцию/seed-миграцию, а не заполняться пользователем.

Текущий справочник:

- `unknown` - запрос не понят или пока не поддерживается.
- `reminder_create` - создать новое напоминание.
- `reminder_need_info` - запрос относится к напоминаниям, но не хватает данных.
- `reminder_list` - показать будущие активные напоминания.
- `reminder_history` - показать историю напоминаний.
- `reminder_delete` - удалить или отменить напоминание.
- `thread_new` - начать новый диалог и перестать учитывать контекст прежней темы.
- `thread_forget` - сбросить текущую тему без удаления истории из БД.
- `web_search` - создать поручение с интернет-поиском.
- `web_search_update` - обновить активное поисковое поручение уточнением.

Для `reminder_delete` backend не удаляет запись по свободному тексту. Он
показывает будущие напоминания с кнопками удаления, чтобы пользователь явно
выбрал нужную запись.

## Контракт ответа

LLM должна вернуть только валидный JSON без Markdown:

```json
{
  "intent": "unknown",
  "confidence": 0.0,
  "reply": null,
  "topic_key": "general",
  "task_text": null,
  "reminder_text": null,
  "due_at": null,
  "event_title": null,
  "event_start": null,
  "event_end": null,
  "attendees": [],
  "needs_clarification": false,
  "clarification_question": null,
  "extracted_context": {}
}
```

Поля `event_*`, `attendees` и `extracted_context` оставлены в контракте, чтобы не
ломать общую схему `IntentResult`, но текущий intent-manager ими не управляет.

## Правила контекста

На текущем reminder-этапе контекст специально ограничен:

- обычный запрос: только текущее сообщение пользователя;
- ответ на уточнение: три сообщения - исходный запрос пользователя, уточняющий
  вопрос ассистента, новый ответ пользователя.

Расширенный контекст включается только если у пользователя есть активная запись
в `user_intent_states` с `intent="reminder_need_info"`. Исходный пользовательский
цепочка сообщений после перехода в pending хранится в
`payload.context_messages`. Старое поле `payload.pending_user_text` остается как
fallback для уже созданных состояний.

Это снижает риск, что LLM начнет фантазировать историю напоминаний на основе
старого диалога. Историю и списки должен отдавать модуль напоминаний из БД.

Следующий этап контекста не должен превращать `user_intent_states` в глобальный
"режим" ассистента. Новые действия должны выбираться intent routing-ом для
каждого сообщения. Например, если пользователь обсуждает поисковую задачу, а
затем пишет "напомни купить его в понедельник", intent-manager должен
распознать `reminder_create`, а backend должен раскрыть "его" через активную
тему или task context.

`recent_context` для обычного сообщения теперь может включать:

- summary активной темы;
- несколько последних сообщений активной темы;
- короткую ссылку на открытую task-сущность, например активную `DelegatedTask`;
- ограниченный фрагмент последнего результата task-сущности, если он нужен для
  явной ссылки на конкретный найденный вариант;
- текущее сообщение пользователя.

Город пользователя хранится в профиле `users.location`, по умолчанию `Москва`,
и передается в web-search prompt. Команда `/location Город` меняет это значение;
это явная настройка профиля, а не intent routing.

Команда `/reset` должна работать как быстрый ручной сброс текущей темы без
обращения к intent-manager. Она эквивалентна фразе "начнем новый диалог", но
обрабатывается на уровне Telegram-входа до LLM.

Чтобы не раздувать контекст и не провоцировать выдуманные напоминания, backend
передает только ограниченную историю и отдельную ссылку на активную задачу. Если
пользователь создает напоминание с явной ссылкой вроде "его", backend после
intent routing запускает отдельный reference-resolver: он получает текущее
сообщение, извлеченный `reminder_text`, активную task-сущность и ограниченный
фрагмент результата поиска, затем возвращает уточненный текст напоминания или
исходный текст без изменений. Backend не распознает местоимения локальными
словарями или регулярными правилами.

Для pending-уточнений `reminder_need_info` этот широкий контекст не используется:
intent-manager получает только исходный запрос, вопрос ассистента и новый ответ
пользователя.

Планируемое разделение ответственности:

- `threads` - контекст разговора, summary и последние сообщения;
- доменные сущности, например `DelegatedTask`, - активная задача и ее
  структурированный контекст;
- `user_intent_states` - только короткое pending-состояние незавершенного
  уточнения, например ожидание времени после вопроса "Во сколько?".

Команда "начнем новый диалог" должна быть ручным сбросом активной темы, а не
обязательным шагом перед каждой новой задачей.

## Состояние пользователя

Текущий pending intent хранится явно в таблице `user_intent_states`.

Основные поля:

- `user_id` - владелец состояния;
- `thread_id` - thread, в котором состояние создано;
- `intent` - pending intent, сейчас используется `reminder_need_info`;
- `status` - `active` или `closed`;
- `payload` - структурированные данные состояния.

Для `reminder_need_info` payload содержит:

```json
{
  "intent": "reminder_need_info",
  "clarification_question": "Во сколько?",
  "pending_user_text": "напомни поесть",
  "context_messages": [
    {"role": "user", "content": "напомни поесть"},
    {"role": "assistant", "content": "Во сколько?"}
  ]
}
```

После успешного `reminder_create`, `reminder_list`, `reminder_history`,
`reminder_delete`, `thread_new`, `thread_forget` или `unknown` активное
состояние пользователя закрывается.

## Поток обработки

1. Telegram webhook передает текст в `AssistantService`.
2. `AssistantService` сохраняет сообщение пользователя.
3. `AssistantService` читает активное состояние пользователя из
   `user_intent_states`.
4. `IntentManager.route()` отправляет в OpenAI текущий запрос и минимальный
   контекст. Если первая классификация возвращает `unknown`, backend повторяет
   запрос с дополнительной подсказкой, чтобы не терять явные поисковые задачи.
5. `AssistantService` исполняет результат:
   - `unknown` - возвращает `reply` или общий текст непонимания;
   - `reminder_need_info` - сохраняет активное состояние и возвращает
     `clarification_question`;
   - `reminder_create` - создает напоминание через `ReminderService`;
   - `reminder_list` - показывает будущие напоминания;
   - `reminder_history` - показывает историю напоминаний;
   - `reminder_delete` - показывает будущие напоминания с кнопками удаления.
   - `thread_new` - создает новую активную тему;
   - `thread_forget` - создает новую активную тему и больше не использует
     контекст прежней темы.
   - `web_search` - создает `DelegatedTask`, запускает web search и сохраняет
     результат в `delegated_tasks.result`;
   - `web_search_update` - обновляет активную `DelegatedTask`, повторно запускает
     web search и сохраняет новый результат.

Для сообщений с изображениями нужен отдельный входной workflow:

- если пришла только картинка без текста, backend сначала задает короткий
  вопрос о том, что нужно сделать, и не обращается к LLM;
- если потом приходит текст, intent-manager получает текст вместе с текущим
  контекстом и только после определения intent backend подключает vision для
  обработки изображения;
- если сразу пришли картинка и текст, backend сначала использует intent-manager
  по тексту, а изображение передает в модель только для того домена, который
  выбрал intent.

## Уточнения

Текст уточняющего вопроса формирует LLM. Вопрос должен быть естественным и
спрашивать только недостающие сведения.

Пример:

```text
Пользователь: Напомни завтра сходить в больницу
Бот: Во сколько?
Пользователь: в 17:00
Бот: Готово. Напомню "сходить в больницу" завтра в 17:00
```

## Напоминания

Для `reminder_create` backend создает запись в `reminders` со статусом
`pending`.

Для `reminder_list` и `reminder_delete` backend выбирает только будущие активные
напоминания:

- `status = "pending"`;
- `due_at > now`;
- сортировка от ближайших к дальним.

Для `reminder_history` backend показывает последние напоминания
пользователя независимо от статуса.

## Fallback без OpenAI API

Если `OPENAI_API_KEY` не задан, `IntentManager` возвращает `unknown`. Локального
эвристического распознавания намерений больше нет.
