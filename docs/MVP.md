# План MVP

## Диалоговый цикл

1. Пользователь пишет или отправляет голосовое в Telegram.
2. Telegram отправляет update в `POST /api/telegram/webhook`.
3. `assistant` находит пользователя и активную тему.
4. LLM intent-manager определяет намерение:
   - `unknown`;
   - `reminder_create`;
   - `reminder_need_info`;
   - `reminder_list`;
   - `reminder_history`;
   - `reminder_delete`.
5. Backend выполняет действие и отправляет ответ через Telegram Bot API.
6. Сообщение пользователя и ответ ассистента сохраняются в БД.

## Контекст

Контекст хранится в таблицах `threads` и `messages`.

- обычный запрос отправляется в intent-manager без старой истории диалога;
- ответ на уточнение отправляется с тремя сообщениями: исходный запрос,
  уточняющий вопрос ассистента и новый ответ пользователя;
- более длинный контекст и summary будут добавлены отдельным этапом.

## Напоминания

LLM извлекает `due_at` в часовом поясе пользователя. Если время неполное,
intent-manager возвращает `reminder_need_info`, а ассистент задает уточнение.
Для "в обед" используется 13:00.

## Google

При первой функции, требующей Google, ассистент отправляет пользователю ссылку OAuth. Токены должны храниться зашифрованно.

## Интернет-поиск

Для MVP поиск выполняет backend или отдельный backend-controlled worker:

- backend хранит цель и контекст;
- backend учитывает `users.location` и команду `/location` при выборе региона поиска;
- поисковый модуль делает поиск и парсинг;
- backend сохраняет результат;
- уточнения обновляют ту же `delegated_task`.

Подробности текущего поведения web-search описаны в [WEB_SEARCH.md](WEB_SEARCH.md).
