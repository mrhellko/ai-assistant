# План разработки AI ассистента

Этот документ - рабочая дорожная карта проекта. В будущих задачах ориентируемся на него: сначала доводим до конца текущий этап, затем переходим к следующему, если пользователь явно не меняет приоритет.

## Цель MVP

Сделать Telegram-ассистента для нескольких пользователей, который:

- принимает текстовые и голосовые сообщения;
- понимает намерения через OpenAI API;
- хранит контекст диалога и отдельных тем в PostgreSQL;
- ставит напоминания и присылает их в Telegram;
- на следующих этапах создает встречи в Google Calendar после авторизации пользователя;
- на следующих этапах выполняет поручения с интернет-поиском и помнит уточнения внутри темы;
- хранит долговременные пользовательские знания в отдельном Memory Module;
- разворачивается на новом VPS одной командой через Docker Compose.

## Архитектурные решения

- Telegram подключается напрямую к backend через webhook FastAPI.
- Backend на FastAPI хранит состояние, решает намерения, управляет доменными сущностями и интеграциями.
- PostgreSQL - единственный источник состояния для пользователей, сообщений, тем, напоминаний, интеграций и поручений.
- `user_intent_states` не является глобальным режимом ассистента. Это только
  короткое pending-состояние для незавершенного уточнения, например "во сколько?"
  после запроса на напоминание.
- Переключение между поиском, напоминаниями, календарем и обычным диалогом
  должно происходить автоматически через intent routing с учетом текущей темы,
  активной task-сущности и безопасно подготовленного контекста.
- Backend не должен распознавать пользовательские смыслы через локальные списки
  ключевых слов, словари фраз или регулярки. Для намерений, местоимений и
  смысловых ссылок используем intent-manager, отдельные LLM-resolver'ы,
  структурированные доменные сущности и явные контракты данных.
- Долговременные знания пользователя должны храниться в отдельной модели Memory Module, а не смешиваться с notes/reminders.
- OpenAI API используется для intent-manager и извлечения структурированных данных.
- Текущий intent-manager покрывает reminder-сценарии; calendar, web-search и
  обычный chat будут добавляться следующими этапами.
- n8n не используем в обязательном runtime MVP: для текущих задач это лишний сервис и лишняя точка отказа.
- Все секреты живут в `.env`, в git хранится только `.env.example`.

## Этап 3. Контекст и новые диалоги

Цель: ассистент должен помнить контекст в рамках текущего диалога, отделять
новые темы и переключать доменные действия без ручного выбора режима.

Пользователь не должен каждый раз писать "начнем новый диалог" перед новой
задачей. Эта команда нужна как явный сброс контекста, а не как обязательный
ритуал. Например, цепочка "найди пруток 20мм" -> "дорого, давай меньше длину"
-> "напомни купить его в понедельник" должна сначала обновить активную
поисковую задачу, а затем создать напоминание, раскрывая "его" через контекст
текущей темы или активной task-сущности.

Задачи:

- Уточнить модель `threads`:
  - активная общая тема;
  - отдельные темы поручений;
  - завершение темы;
  - ручной сброс темы без потери истории в БД.
- Зафиксировать разделение ответственности:
  - `threads` хранят контекст разговора;
  - `DelegatedTask` и будущие task-сущности хранят активную задачу и ее
    структурированный контекст;
  - `user_intent_states` хранит только pending-уточнение для незавершенного
    действия.
- Добавить слой подготовки контекста для intent-manager:
  - [x] текущее сообщение;
  - [x] summary активной темы;
  - [x] последние сообщения активной темы;
  - [x] короткие ссылки на активные task/entity references, например "пруток 20мм".
- Научить backend безопасно раскрывать местоимения и ссылки вроде "его",
  "этот вариант", "ту встречу" через отдельный reference-resolver, который
  получает текущий текст, `reminder_text` и активную task-сущность без локальных
  словарей местоимений в backend.
- Добавить команды естественным языком:
  - [x] "начнем новый диалог";
  - [x] "забудь эту тему";
  - "вернемся к прошлой задаче" как отдельная задача после MVP, если понадобится.
- Добавить summary длинных диалогов, чтобы не отправлять в LLM слишком много сообщений.
- Ограничить контекст последними сообщениями плюс summary.
- Добавить тестовые сценарии переключения действий:
  - web-search уточнение продолжает активную поисковую задачу;
  - reminder-команда внутри поисковой темы создает напоминание и использует
    предмет из контекста;
  - "начнем новый диалог" отключает влияние прежней темы на новые сообщения.

Критерий готовности:

- Обычный разговор сохраняет контекст.
- После "начнем новый диалог" старый контекст не влияет на ответ.
- Уточнение внутри поручения не теряет исходный предмет обсуждения.
- Команда другого домена внутри текущей темы переключает действие без потери
  ссылок на предмет обсуждения, например "напомни купить его в понедельник"
  после поиска прутка создает напоминание про этот пруток, а не текст
  "купить его".

## Этап 4. Google OAuth и календарь

Цель: ассистент должен создавать встречи в Google Calendar и использовать контакты.

Задачи:

- Реализовать OAuth callback полностью:
  - обмен `code` на tokens;
  - сохранение refresh token;
  - обновление access token;
  - обработка ошибок авторизации.
- Зашифровать payload токенов перед записью в БД.
- Подключить Google Calendar API.
- Подключить Google People API для поиска контактов.
- На первой календарной операции просить пользователя подключить Google.
- Извлекать из естественного языка:
  - название встречи;
  - дату и время начала;
  - длительность или время окончания;
  - участников;
  - описание.

Критерий готовности:

- "создай встречу с Иваном завтра в 15:00 на час" просит Google OAuth, если токенов нет.
- После авторизации та же команда создает событие в календаре.
- Если контакт неоднозначный, ассистент задает уточняющий вопрос.

## Правила разработки

- Любая новая функция должна сохранять состояние в PostgreSQL, если ей нужен контекст или повторное выполнение.
- Перед изменениями проверяем, не ломается ли команда запуска из README.
- Перед push проверяем хотя бы синтаксис Python и `git status`.
- Не коммитим реальные `.env`, токены, cookies, OAuth payload и дампы БД.
- Если появляется выбор между внешним workflow и backend-кодом, бизнес-логику кладем в backend.
