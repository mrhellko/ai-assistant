# План разработки AI ассистента

Этот документ - рабочая дорожная карта проекта. В будущих задачах ориентируемся на него: сначала доводим до конца текущий этап, затем переходим к следующему, если пользователь явно не меняет приоритет.

## Цель MVP

Сделать Telegram-ассистента для нескольких пользователей, который:

- принимает текстовые и голосовые сообщения;
- понимает намерения через OpenAI API;
- хранит контекст диалога и отдельных тем в PostgreSQL;
- ставит напоминания и присылает их в Telegram;
- создает встречи в Google Calendar после авторизации пользователя;
- выполняет поручения с интернет-поиском и помнит уточнения внутри темы;
- разворачивается на новом VPS одной командой через Docker Compose.

## Архитектурные решения

- Telegram подключается напрямую к backend через webhook FastAPI.
- Backend на FastAPI хранит состояние, решает намерения, управляет доменными сущностями и интеграциями.
- PostgreSQL - единственный источник состояния для пользователей, сообщений, тем, напоминаний, интеграций и поручений.
- OpenAI API используется для intent routing, извлечения структурированных данных, обычного диалога и анализа результатов поиска.
- n8n не используем в обязательном runtime MVP: для текущих задач это лишний сервис и лишняя точка отказа.
- Все секреты живут в `.env`, в git хранится только `.env.example`.

## Этап 0. Базовый каркас

Статус: сделано.

Что сделано:

- Docker Compose для `assistant` и `postgres`.
- Caddy reverse proxy для HTTPS.
- FastAPI backend.
- SQLAlchemy-модели для пользователей, тем, сообщений, напоминаний, токенов интеграций и поручений.
- LLM intent router.
- Заготовки Google Calendar.
- Telegram webhook.
- Reminder loop.
- Базовая документация.

Критерий готовности этапа:

- `docker compose up -d --build` поднимает все сервисы.
- `/health` backend отвечает `{"status":"ok"}`.
- Ветка `master` в GitHub содержит актуальное состояние.

## Этап 3. Контекст и новые диалоги

Цель: ассистент должен помнить контекст в рамках текущего диалога и отделять новые темы.

Задачи:

- Уточнить модель `threads`: активная общая тема, отдельные темы поручений, завершение темы.
- Добавить команды естественным языком:
  - "начнем новый диалог";
  - "забудь эту тему";
  - "вернемся к прошлой задаче" как отдельная задача после MVP, если понадобится.
- Добавить summary длинных диалогов, чтобы не отправлять в LLM слишком много сообщений.
- Ограничить контекст последними сообщениями плюс summary.

Критерий готовности:

- Обычный разговор сохраняет контекст.
- После "начнем новый диалог" старый контекст не влияет на ответ.
- Уточнение внутри поручения не теряет исходный предмет обсуждения.

## Этап 4. Google OAuth и календарь

Цель: ассистент должен создавать встречи в Google Calendar и использовать контакты.

Задачи:

- Реализовать OAuth callback полностью:
  - обмен `code` на tokens;
  - сохранение refresh token;
  - обновление access token;
  - обработка ошибок авторизации.
- Зашифровать payload токенов перед записью в БД.
- Подключить Google Calendar API.
- Подключить Google People API для поиска контактов.
- На первой календарной операции просить пользователя подключить Google.
- Извлекать из естественного языка:
  - название встречи;
  - дату и время начала;
  - длительность или время окончания;
  - участников;
  - описание.

Критерий готовности:

- "создай встречу с Иваном завтра в 15:00 на час" просит Google OAuth, если токенов нет.
- После авторизации та же команда создает событие в календаре.
- Если контакт неоднозначный, ассистент задает уточняющий вопрос.

## Этап 5. Поручения с интернет-поиском

Цель: ассистент должен выполнять поисковые поручения и сохранять контекст уточнений.

Задачи:

- Определить backend contract для web-search задач.
- Хранить `DelegatedTask` с objective, context, status, result.
- Реализовать модуль поиска:
  - выполнить web search;
  - собрать варианты;
  - нормализовать цену, наличие, доставку, ссылки;
  - сохранить результат в backend.
- Добавить LLM-анализ результатов:
  - сравнить варианты;
  - объяснить оптимальный выбор;
  - отметить риски;
  - предложить уточнения, если данных мало.
- Поддержать уточнения:
  - "дорого";
  - "давай длиной 200мм";
  - "только с доставкой";
  - "ищи в Москве".

Критерий готовности:

- "найди где купить пруток сталь 20 диаметром 40мм длиной 500 мм" создает задачу поиска.
- "дорого, давай длиной 200мм" обновляет ту же задачу, сохраняя материал и диаметр.
- Пользователь получает список вариантов с ценами, ссылками и рекомендацией.

## Этап 6. Голосовые сообщения

Цель: голосовые сообщения должны работать как текстовые.

Задачи:

- В backend скачать voice file из Telegram.
- Транскрибировать через OpenAI audio transcription.
- Обработать текст как обычное сообщение.
- Сохранить в `raw` ссылку или metadata исходного voice message.
- Ответ отправлять текстом. Голосовой ответ отложить до отдельного этапа.

Критерий готовности:

- Пользователь отправляет голосовое "напомни завтра в обед принять таблетку".
- Ассистент создает то же напоминание, что и по тексту.

## Правила разработки

- Любая новая функция должна сохранять состояние в PostgreSQL, если ей нужен контекст или повторное выполнение.
- Перед изменениями проверяем, не ломается ли команда запуска из README.
- Перед push проверяем хотя бы синтаксис Python и `git status`.
- Не коммитим реальные `.env`, токены, cookies, OAuth payload и дампы БД.
- Если появляется выбор между внешним workflow и backend-кодом, бизнес-логику кладем в backend.
